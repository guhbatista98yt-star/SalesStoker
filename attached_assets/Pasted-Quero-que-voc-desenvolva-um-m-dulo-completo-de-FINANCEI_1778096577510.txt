2. SQL REAL DE CONTAS A RECEBER
====================================================

Usar o SQL abaixo como fonte principal dos dados do módulo.

Antes da consulta no DB2, executar:

SET CURRENT SCHEMA DBA;

SQL principal:

SET CURRENT SCHEMA DBA; 

SELECT
    CRSV.IDEMPRESA,

    CF.IDVENDEDOR AS IDVENDEDOR_CLIENTE,
    VEND.NOME AS NOME_VENDEDOR,

    CF.IDCLIFOR,
    CF.NOME AS NOME_CLIENTE,

    CRSV.IDTITULO,
    CRSV.DIGITOTITULO,
    CRSV.SERIENOTA,

    CASE 
        WHEN COALESCE(N.NUMCUPOMFISCAL, 0) = 0 THEN N.NUMNOTA
        ELSE N.NUMCUPOMFISCAL
    END AS NUMNOTA,

    CRSV.IDPLANILHA,

    CRSV.DTMOVIMENTO,
    CRSV.DTVENCIMENTO,
    CRSV.DTULTIMOPAGAMENTO,

    CASE 
        WHEN CRSV.DTVENCIMENTO < CURRENT DATE THEN
            DAYS(CURRENT DATE) - DAYS(CRSV.DTVENCIMENTO)
        ELSE 0
    END AS DIAS_VENCIDO,

    DECIMAL(COALESCE(CRSV.VALTITULO, 0), 15, 2) AS VALOR_TITULO,
    DECIMAL(COALESCE(CRSV.VALSALDOTITULO, 0), 15, 2) AS VALOR_SALDO_TITULO,
    DECIMAL(COALESCE(CRSV.VALLIQUIDOTITULO, 0), 15, 2) AS VALOR_LIQUIDO_TITULO,

    DECIMAL(
        COALESCE(CRSV.SUMVALJUROSMORA, 0) +
        COALESCE(CRSV.SUMVALJUROSPOSTERGADO, 0),
        15, 2
    ) AS VALOR_JUROS_PENDENTE,

    DECIMAL(COALESCE(CRSV.SUMVALDESCCONCEDIDOTITULO, 0), 15, 2) AS VALOR_DESCONTO_CONCEDIDO,
    DECIMAL(COALESCE(CRSV.SUMVALPAGAMENTOTITULO, 0), 15, 2) AS VALOR_PAGO,

    FPR.DESCRRECEBIMENTO AS FORMA_RECEBIMENTO,

    CRSV.ORIGEMMOVIMENTO,
    CRSV.OBSTITULO,

    CF.ENDERCobranca AS ENDERECO_COBRANCA,
    CF.BAIRROCOBRANCA AS BAIRRO_COBRANCA,
    CI.DESCRCIDADE AS CIDADE_COBRANCA,
    CI.UF AS UF_COBRANCA

FROM
    CONTAS_RECEBER_SALDOS_VIEW CRSV

JOIN CONTAS_RECEBER CR ON
    CR.IDEMPRESA = CRSV.IDEMPRESA
    AND CR.IDCLIFOR = CRSV.IDCLIFOR
    AND CR.IDTITULO = CRSV.IDTITULO
    AND CR.DIGITOTITULO = CRSV.DIGITOTITULO
    AND CR.SERIENOTA = CRSV.SERIENOTA

JOIN CLIENTE_FORNECEDOR CF ON
    CF.IDCLIFOR = CRSV.IDCLIFOR

LEFT JOIN CLIENTE_FORNECEDOR VEND ON
    VEND.IDCLIFOR = CF.IDVENDEDOR

JOIN FORMA_PAGREC FPR ON
    FPR.IDRECEBIMENTO = CRSV.IDRECEBIMENTO

LEFT JOIN NOTAS N ON
    N.IDEMPRESA = CRSV.IDEMPRESA
    AND N.IDPLANILHA = CRSV.IDPLANILHA

LEFT JOIN CIDADES_IBGE CI ON
    CI.IDCIDADE = CF.IDCIDADECOBRANCA

WHERE
    COALESCE(CRSV.VALTITULO, 0) <> 0

    AND (
        COALESCE(CRSV.VALSALDOTITULO, 0) +
        COALESCE(CRSV.VALLIQUIDOTITULO, 0)
    ) > 0

    AND COALESCE(FPR.FLAGCHEQUE, 'F') <> 'T'

    AND CRSV.DTMOVIMENTO >= DATE('1900-01-01')
    AND CRSV.DTMOVIMENTO <= CURRENT DATE

    AND (
        (
            CRSV.FLAGCARTAOPROPRIO = 'T'
            AND CRSV.ORIGEMMOVIMENTO <> 'CTP'
            AND TRIM(COALESCE(CRSV.OBSTITULO, '')) <> 'Débito Fatura Anterior'
            AND NOT EXISTS (
                SELECT
                    1
                FROM
                    CONTAS_RECEBER CR2
                WHERE
                    CR2.IDPLANILHA IN (
                        SELECT
                            CRB.IDPLANILHA
                        FROM
                            CONTAS_RECEBER_BAIXAS CRB
                        WHERE
                            CRB.IDTITULO = CRSV.IDTITULO
                            AND CRB.IDEMPRESA = CRSV.IDEMPRESA
                            AND CRB.DIGITOTITULO = CRSV.DIGITOTITULO
                            AND CRB.IDCLIFOR = CRSV.IDCLIFOR
                            AND CRB.SERIENOTA = CRSV.SERIENOTA
                    )
                    AND CR2.ORIGEMMOVIMENTO = 'CTP'
            )
        )

        OR (
            CRSV.FLAGCARTAOPROPRIO = 'T'
            AND CRSV.ORIGEMMOVIMENTO = 'CTP'
        )

        OR (
            CRSV.FLAGCARTAOPROPRIO = 'F'
            AND (
                NOT (
                    COALESCE(CRSV.IDADMINISTRADORA, 0) > 0
                    AND CRSV.ORIGEMMOVIMENTO = 'CAI'
                )
                OR CR.FLAGAGRUPADO = 'T'
            )
        )
    )

ORDER BY
    VEND.NOME,
    CF.NOME,
    CRSV.DTVENCIMENTO,
    CRSV.IDTITULO

FOR READ ONLY;

Observação:
Se a coluna CF.ENDERCobranca gerar erro por causa do nome no DB2, ajustar para o nome real existente no banco, sem quebrar o restante da estrutura.

====================================================
4. REGRA PRINCIPAL DOS VALORES
====================================================

Usar como verdade principal do valor em aberto:

VALOR_SALDO_TITULO = valor_aberto

VALOR_TITULO é o valor original do título.
VALOR_PAGO é o valor já pago.
VALOR_LIQUIDO_TITULO deve ser exibido apenas como informação adicional.
VALOR_LIQUIDO_TITULO não deve substituir VALOR_SALDO_TITULO nos cards, relatórios, rankings ou cobrança.

Para cards, rankings, relatórios e gestão de cobrança, usar sempre:

valor_aberto = VALOR_SALDO_TITULO

Quando for necessário exibir valor com juros:

valor_total_com_juros = valor_aberto + valor_juros_pendente

A soma usada no WHERE da SQL:
VALSALDOTITULO + VALLIQUIDOTITULO
serve apenas como critério de filtro da própria consulta.
Não usar essa soma como valor gerencial do contas a receber.

====================================================
6. STATUS FINANCEIRO CALCULADO
====================================================

A SQL já traz DIAS_VENCIDO, mas o sistema deve calcular status visual com base em DTVENCIMENTO e valor_aberto.

Regras:

Se valor_aberto <= 0:
STATUS = RECEBIDO

Se valor_aberto > 0 e dtvencimento < hoje:
STATUS = VENCIDO

Se valor_aberto > 0 e dtvencimento = hoje:
STATUS = VENCE_HOJE

Se valor_aberto > 0 e dtvencimento > hoje:
STATUS = A_VENCER

Campo dias_atraso:
Usar DIAS_VENCIDO vindo da SQL.
Se vier nulo, recalcular no backend.

Status financeiro do cliente:

EM_DIA:
Cliente não possui títulos vencidos.

ATENCAO:
Cliente possui título vencendo hoje ou atraso de até 7 dias.

ATRASADO:
Cliente possui atraso entre 8 e 30 dias.

CRITICO:
Cliente possui atraso acima de 30 dias ou valor vencido muito alto.

Essas faixas devem ficar fáceis de ajustar futuramente em configuração.

8. SINCRONIZAÇÃO COM ERP
====================================================

Criar endpoint:

POST /api/admin/sync/contas-receber

Se o projeto já usa X-Admin-Token, proteger esse endpoint com o mesmo token.

Fluxo:

1. Conectar no DB2/ERP.
2. Executar SET CURRENT SCHEMA DBA.
3. Executar o SQL real de contas a receber.
4. Receber os dados.
5. Normalizar nomes dos campos.
6. Criar chave_titulo.
7. Calcular status.
8. Validar datas e valores.
9. Atualizar cache_contas_receber.
10. Não duplicar títulos.
11. Registrar data/hora da última sincronização.
12. Retornar resumo da sincronização.

Retorno esperado:

{
  "success": true,
  "registros_importados": 0,
  "clientes": 0,
  "valor_total_aberto": 0.00,
  "valor_total_vencido": 0.00,
  "ultima_atualizacao": "DD/MM/YYYY HH:mm:ss"
}

Caso a SQL não retorne dados, não quebrar o sistema.
Mostrar:

“Nenhum título em aberto encontrado no contas a receber.”

Caso falhe a conexão com ERP, mostrar:

“Não foi possível atualizar agora. Verifique a conexão com o ERP.”

Além de visualizar contas a receber, o módulo precisa ajudar na gestão prática de cobrança.

Criar uma camada interna de acompanhamento financeiro, sem alterar os dados do ERP.

Permitir registrar:

- Status interno da cobrança
- Observação interna
- Última cobrança realizada
- Próxima data de cobrança
- Responsável pela cobrança
- Promessa de pagamento
- Data prometida para pagamento
- Motivo do atraso
- Canal de contato usado
- Histórico de interações

Status interno sugerido:

- A COBRAR
- COBRADO
- EM NEGOCIAÇÃO
- PROMETEU PAGAR
- AGUARDANDO RETORNO
- PROBLEMA COMERCIAL
- PROBLEMA DE ENTREGA
- PROBLEMA DE NOTA FISCAL
- ENVIAR PARA GESTOR
- JURÍDICO
- RESOLVIDO

Esses status são internos do sistema e não devem alterar o ERP.

Criar tabela:

CREATE TABLE IF NOT EXISTS financeiro_cobrancas (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    chave_titulo TEXT,
    idempresa INTEGER,
    idclifor INTEGER,
    idtitulo INTEGER,
    digitotitulo TEXT,
    serienota TEXT,
    status_interno TEXT,
    observacao TEXT,
    motivo_atraso TEXT,
    canal_contato TEXT,
    responsavel TEXT,
    data_cobranca TEXT,
    proxima_acao TEXT,
    data_proxima_acao TEXT,
    promessa_pagamento TEXT,
    data_promessa_pagamento TEXT,
    criado_em TEXT,
    atualizado_em TEXT,
    usuario TEXT
);

Criar histórico:

CREATE TABLE IF NOT EXISTS financeiro_cobrancas_historico (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    id_cobranca INTEGER,

    digitotitulo TEXT,
    serienota TEXT,

    valor_anterior TEXT,
    valor_novo TEXT,
    usuario TEXT,
    criado_em TEXT
);

19. FILA INTELIGENTE DE COBRANÇA
====================================================

Criar uma tela/seção chamada:

Fila de Cobrança

Ela deve listar automaticamente os clientes/títulos que precisam de atenção.

Critérios de prioridade:

1. Maior valor vencido
2. Maior quantidade de dias em atraso
3. Promessa de pagamento vencida
4. Cliente sem cobrança registrada
5. Cliente crítico acima de 30 dias
6. Cliente com maior quantidade de títulos vencidos
7. Cliente com juros pendente

Cada item da fila deve mostrar:

- Prioridade
- Cliente
- Vendedor
- Total vencido
- Juros pendente
- Título mais antigo
- Maior atraso
- Última cobrança
- Próxima ação
- Status interno
- Botão Registrar cobrança
- Botão Ver cliente

Prioridade sugerida:

ALTA:
- Valor vencido alto
- Atraso acima de 30 dias
- Promessa de pagamento vencida

MÉDIA:
- Atraso entre 8 e 30 dias

BAIXA:
- Vence hoje ou atraso até 7 dias

20. DETALHE DE COBRANÇA DO CLIENTE
====================================================

No detalhe do cliente, adicionar aba/seção:

Cobrança

Mostrar:

- Histórico de contatos
- Observações internas
- Status atual da cobrança
- Próxima ação
- Data da próxima ação
- Promessa de pagamento
- Responsável pelo acompanhamento

Criar botão:

Registrar nova cobrança

Formulário:

- Status interno
- Canal de contato: Telefone, WhatsApp, E-mail, Presencial, Outro
- Observação
- Motivo do atraso
- Promessa de pagamento: Sim/Não
- Data prometida
- Próxima ação
- Data da próxima ação

Exemplo de uso:
Cliente está com R$ 8.500,00 vencido há 22 dias.
Financeiro liga.
Cliente promete pagar dia 15.
Usuário registra:
Status interno: PROMETEU PAGAR
Canal: Telefone
Promessa de pagamento: Sim
Data prometida: 15/05/2026
Próxima ação: Conferir pagamento
Data próxima ação: 16/05/2026

Se a data prometida passar e continuar em aberto, destacar esse cliente na fila de cobrança.

21. INDICADORES GERENCIAIS
====================================================

Adicionar indicadores:

- Total em aberto
- Total vencido
- Percentual inadimplente
- Quantidade de clientes inadimplentes
- Valor vencido por vendedor
- Ranking de maiores devedores
- Ranking de títulos mais antigos
- Valor vencido por faixa de atraso
- Valor com promessa de pagamento
- Promessas de pagamento vencidas
- Clientes sem cobrança registrada
- Clientes cobrados nos últimos 7 dias
- Clientes que precisam de nova ação hoje
- Juros pendente total
- Juros pendente por cliente
- Juros pendente por vendedor

Fórmulas:

Total em aberto:
Soma valor_aberto onde valor_aberto > 0.

Total vencido:
Soma valor_aberto onde status = VENCIDO.

Vence hoje:
Soma valor_aberto onde status = VENCE_HOJE.

A vencer:
Soma valor_aberto onde status = A_VENCER.

Clientes com pendência:
Quantidade distinta de idclifor com valor_aberto > 0.

Clientes vencidos:
Quantidade distinta de idclifor com status = VENCIDO.

Maior atraso:
Maior dias_atraso entre títulos abertos.

Percentual inadimplente:
total_vencido / total_em_aberto * 100.
